# Race Capture — дизайн (частина 2: автоматизоване внесення результатів гонок)

Дата: 2026-06-11
Статус: затверджено користувачем (брейншторм-сесія)

## Мета

Прибрати ручне внесення результатів гонок Gauntlet-челенджа. Гравець під час/після
челенджа натискає глобальну гарячу клавішу — застосунок робить скріншот, розпізнає
екран гри, витягує доступний шматок даних і поступово наповнює «сесію челенджа»
(5 гонок). Оверлей показує прогрес, не перериваючи гру. Наприкінці гравець
переглядає зібрані дані в застосунку, виправляє і зберігає одним натисканням.

## Контекст і обмеження

- Гра: Asphalt 9 (ALU), Microsoft Store, режим borderless fullscreen,
  десктоп 2560×1600 (16:10). Застосунок має працювати і на інших пристроях:
  інші роздільні здатності та співвідношення сторін. Розпізнавання повинно
  допускати невеликі відхилення позицій елементів (запас у ROI, скан із
  невеликими зсувами, самоперевірка через OCR-якорі), а не вимагати
  попіксельної точності.
- У БД зберігаються **тільки власні гонки**: трек, авто, ранг, час
  (наявна модель `Race`, шлях збереження `RacesService.save()`).
- Дані суперника не зберігаються.
- Сесія челенджа живе **тільки в пам'яті** (без персистенції у файл).
- Текст на екранах — обмежений словник: назви авто і треків уже є в БД,
  іконки карт — у `resources/icons/maps/`.

## Підхід до розпізнавання (обраний: фіксовані регіони + словник)

Ігровий UI детермінований, тому замість динамічного пошуку контурів
(поточний `recognition.py`, який ламається) — детермінований пайплайн:

1. **Класифікація екрана**: кожен екстрактор має дешевий `matches(img) → score`
   по якірних ознаках (кольори/текст у фіксованих точках, структура лейауту).
   Движок обирає екстрактор з найвищим score; нижче порога — «екран не розпізнано».
2. **ROI у відносних координатах** (частки ширини/висоти кадру) — масштабується
   на будь-яку роздільну здатність. Для стійкості до інших співвідношень сторін
   і дрібних відмінностей лейауту: ROI задаються з запасом (~10–15%), а пошук
   якірних елементів (заголовок "RACE N") сканується з кількома невеликими
   зсувами; знайдений зсув застосовується до решти ROI панелі.
3. **OCR вузькими шматками** (Tesseract): кроп ROI + апскейл + бінаризація;
   whitelist під тип поля — час: `0123456789:.`, ранг: `0123456789,`,
   назви: літери+пробіл.
4. **Словникове зіставлення**: OCR-результат назв fuzzy-матчиться (rapidfuzz)
   проти назв авто/треків з БД, порівняння без пробілів (знімає проблему
   злиплих пробілів). На виході — топ-кандидати зі score.
5. **Іконки карт**: `cv2.matchTemplate` проти `resources/icons/maps/`
   (потрібно для майбутніх екранів, де треку немає, а є іконка карти).

### Екран-акордеон (CAR SELECTION) — два варіанти

Уточнено на реальних скріншотах (2026-06-12). Розгорнуту панель знаходимо
самоперевіркою заголовка "RACE N" по 5 позиціях сітки.

- **before** (гонка ще не проїхана): зліва авто суперника + ранг, по центру
  трек, праворуч SELECT CAR, внизу зліва TIME TO BEAT. Витягуємо ЛИШЕ трек.
- **after** (гонка проїхана): додатково праворуч авто гравця + ранг, внизу
  праворуч YOUR TIME (жовтий = WON, червоний = LOST). Витягуємо трек +
  авто/ранг/час гравця.

Ліву частину (суперник, TIME TO BEAT) не читаємо ніколи — дані суперника не
повинні зберігатися як власні гонки. Варіант визначається за вмістом
(читабельні YOUR TIME або авто гравця праворуч). Нюанси: назва треку в
центрі — рухомий рядок (marquee); розпізнається через `TrackResolver`:
статична назва карти звужує кандидатів до 2–3 треків карти, а фрагмент
будь-якої фази прокрутки матчиться partial_ratio проти ПОДВОЄНОЇ
нормалізованої назви треку (циклічний підрядок). Один трек на карті →
повертається одразу; нерозрізнювані кандидати → найкращий зі зниженим
score + усі кандидати (жовта підсвітка в рев'ю, уточнюється наступним
захопленням через злиття сесії). Червоний час LOST потребує max-channel
препроцесингу в `read_time`. Після всіх 5 гонок
замість акордеона гра показує екран CHALLENGE WON/LOST (без назв треків) —
його екстрактор у списку розширень.

Відхилені альтернативи:
- Динамічний пошук контурів (поточний код) — тендітний, незрозуміло який бокс що містить.
- Локальна VLM (Gemma/Ollama) — GPU зайнятий грою, важка залежність, ризик
  галюцинацій у часах. Можливий опціональний бекенд у майбутньому — інтерфейс
  екстракторів цього не виключає.

## Архітектура

```
alu_gauntlet_helper/
├── capture/
│   ├── hotkey.py                 # GlobalHotkeyService — глобальний хоткей (lib keyboard)
│   ├── screen_grab.py            # знімок екрана через mss
│   └── capture_controller.py     # оркестрація: хоткей → скрін → розпізнавання → сесія → оверлей
├── screen_recognition/           # переробка наявного модуля
│   ├── engine.py                 # RecognitionEngine: класифікувати екран → викликати екстрактор
│   ├── screens/
│   │   ├── base.py               # ScreenExtractor: matches(img) → score, extract(img) → list[RaceCapture]
│   │   └── challenge_accordion.py  # v1: екран-акордеон після челенджа
│   ├── ocr.py                    # обгортки Tesseract: read_time(), read_rank(), read_name()
│   ├── matching.py               # VocabularyMatcher (rapidfuzz) + MapIconMatcher
│   └── regions.py                # ROI-карти у відносних координатах
├── services/
│   └── challenge_session.py      # ChallengeSessionService: стан 5 гонок, злиття часткових даних
└── views/
    ├── overlay.py                # OverlayWindow: click-through панель статусу
    └── capture_review_dialog.py  # рев'ю 5 гонок перед записом у БД
```

Потік даних: хоткей (потік lib keyboard) → Qt-сигнал → скрін (mss) →
RecognitionEngine у фоновому QThread → `RaceCapture` →
`ChallengeSessionService.merge()` → сигнал → оверлей оновлює статус.
Збереження — через рев'ю-діалог → наявний `RacesService.save()`.

### Моделі

```python
class FieldGuess(BaseModel):
    value: int             # id треку/авто (найкращий кандидат)
    score: float           # впевненість 0..1
    candidates: list[tuple[int, float]]  # топ-кандидати (id, score)

class RaceCapture(BaseModel):
    race_number: int       # 1..5
    track: FieldGuess | None
    car: FieldGuess | None
    rank: int | None
    time: int | None       # мс, як у Race.time
    source_screen: str
```

`ChallengeSession` тримає `dict[race_number → RaceCapture]`.
Правило злиття: поле перезаписується, лише якщо нове значення впевненіше
або старе порожнє.

## Сесія челенджа

- Створюється автоматично при першому успішному захопленні.
- Повторний скрін тієї ж гонки уточнює/перезаписує поля за правилом упевненості.
- Повна, коли всі 5 гонок мають трек + авто + час. Ранг — бажаний;
  якщо не зчитався, у рев'ю підставляється поточний ранг авто з БД.
- Живе тільки в пам'яті.
- «Save» у рев'ю записує гонки в БД і очищує сесію; «Discard» очищує.
- Захоплення, що конфліктує з повною сесією → підказка в оверлеї
  «сесія вже повна — збережи або скинь».

## Захоплення

- Глобальний хоткей, дефолт F8 (налаштовується в Settings). F9 — показати/сховати оверлей.
- Скрін через mss, основний монітор (вибір монітора в Settings).
  Перед знімком оверлей ховається на ~50 мс.
- Розпізнавання у фоновому QThread; повторне натискання під час обробки ігнорується.
- Кожен скрін опціонально зберігається у `data/captures/` (ротація, останні ~20) —
  дебаг + джерело фікстур.
- Шлях до tesseract.exe — з хардкоду в Settings, з автопошуком типових шляхів.

## Оверлей

- Маленька frameless-панель у куті (кут налаштовується), завжди зверху,
  click-through (`WindowTransparentForInput`).
- Вміст: 5 рядків стану гонок (✓ / ⚠ непевне / — немає даних) + рядок останньої
  події + підказка хоткеїв.
- Тільки відображення, без взаємодії. З'являється автоматично при першому
  захопленні, якщо був прихований.

## Рев'ю та збереження

- Діалог у застосунку: 5 рядків; трек і авто через наявний `ItemCompleter`
  (передзаповнені найкращим кандидатом), ранг і час — поля вводу.
- Низький confidence — жовта підсвітка, порожнє обов'язкове поле — червона.
- Біля кожного рядка — мініатюра вирізаного шматка скріншота-джерела.
- «Save all» зберігає всі валідні рядки через `RacesService.save()`
  (оновлення рангу авто спрацьовує як у ручній модалці).

## Обробка помилок

- Екран не розпізнано → оверлей «Екран не розпізнано», сесія не змінюється.
- Поле не зчиталося (порожній OCR / score нижче порога) → поле пусте, ⚠ в оверлеї;
  виправлення повторним скріном або вручну в рев'ю.
- Tesseract не знайдено → статус у Settings + повідомлення при першій спробі
  захоплення, поле для шляху.
- Хоткей не зареєструвався → повідомлення в Settings, вибір іншої клавіші.
- Винятки в потоці розпізнавання не валять застосунок: лог + «Помилка
  розпізнавання» в оверлеї.

## Тестування

- Юніт-тести екстракторів на фікстурах: наявні зменшені скріншоти у
  `tests/fixtures/` (+ згодом реальні скріни з `data/captures/`) →
  очікуваний `RaceCapture`. Головний захист від регресій при калібруванні ROI.
- Тести `VocabularyMatcher` на типових OCR-помилках (злиплі пробіли, зайві
  символи → правильний кандидат).
- Тести злиття сесії (часткові дані з різних скрінів, перезапис за впевненістю).
- Ручна перевірка end-to-end з грою.

## Калібрування без повнорозмірних скріншотів

Користувач не може надати повнорозмірні скріншоти. Тому:

- Первинне калібрування ROI — на наявних зменшених прикладах (600×375, 16:10),
  відносні координати від роздільної здатності не залежать.
- Точність добирається ітеративно у реальній грі: опція `save_captures`
  зберігає скріни захоплень у `data/captures/`, які стають фікстурами
  для тестів і подальшого підстроювання.
- Дизайн розпізнавання розрахований на неточність: запас у ROI, скан зсувів,
  словникове зіставлення (виправляє часткові OCR-помилки), рев'ю-діалог
  (людина підтверджує перед записом у БД).

## Порядок розширення (поза v1)

1. Екстрактор екрана результату одразу після гонки (зручніше скринити під час гри).
2. Екстрактор екрана CHALLENGE WON (всі 5 авто+часи одним скріном; трек — через
   іконку карти + підтвердження).
3. Екстрактори setup-екранів (треки до початку гонок).
4. Опціональний VLM-бекенд як альтернатива OCR.

## Ескіз частини 1 (вибір суперника) — окремий майбутній брейншторм

- Те саме ядро розпізнавання: екстрактори екранів суперника
  (треки+ранги без часів; іконки карт+часи+ранги; setup обраного суперника).
- Оцінка: часи суперника проти власних середніх часів по треках
  (`get_car_stats_for_track` вже є). Для іконки карти без точного треку —
  розглядаються всі треки карти, час суперника підказує найімовірніший.
- UI: розвиток вкладки Car Selection — хоткей захоплює суперника, треки
  автозаповнюються, поруч найкращі авто та вердикт
  «битий / ризиковано / не битий» по кожному треку.
