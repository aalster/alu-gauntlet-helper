# Дизайн: інсталятор для ALU Gauntlet Helper

Дата: 2026-06-12
Статус: затверджено (варіант PyInstaller `--onedir` + Inno Setup)

## Мета

Замінити поточну збірку PyInstaller `--onefile` (один великий .exe, повільний старт через розпакування в temp) на класичну інсталяцію через сетап-файл: швидкий запуск, ярлики, деінсталятор, оновлення поверх старої версії.

## Рішення

- **Збірка**: PyInstaller `--onedir` — тека з `ALU Gauntlet Helper.exe` + `_internal/`.
- **Інсталятор**: Inno Setup, скрипт `installer/setup.iss` у репозиторії; результат — один `ALU-Gauntlet-Helper-Setup-<версія>.exe`.
- **Тип інсталяції**: per-user, без UAC (`PrivilegesRequired=lowest`), тека `{localappdata}\Programs\ALU Gauntlet Helper`.
- **Дистрибуція**: GitHub Releases, оновлення вручну (новий сетап ставиться поверх старого).
- **Tesseract**: портативна копія (`tesseract.exe` + DLL + `eng.traineddata`) кладеться в `installer/tesseract/` (не в git) і пакується інсталятором у підтеку `tesseract/` поруч із застосунком.

## Зміни в коді

### 1. Шляхи даних користувача

Зараз `app.db` та `data/` пишуться у поточну робочу директорію (cwd) — з IDE це корінь проєкту, але при запуску встановленого exe через ярлик cwd непередбачувана.

Рішення: дані живуть **поруч з exe** (per-user інсталяція в `%LOCALAPPDATA%\Programs` доступна на запис). На самому початку `main()` у frozen-режимі процес переходить у теку exe:

```python
if getattr(sys, "frozen", False):
    os.chdir(os.path.dirname(sys.executable))
```

Після цього весь наявний код (`app.db`, `data/captures`, `data/maps`, `data/cars`, відносні шляхи іконок у БД) працює без змін незалежно від способу запуску. У dev-режимі поведінка не змінюється. Інваріант «cwd = тека даних, не змінювати cwd після старту» фіксується коментарем.

Налаштування зберігаються в таблиці `settings` тієї самої `app.db`, тому окремих змін не потребують.

### 2. Міграція старих даних

Не потрібна: єдиний користувач один раз вручну копіює `app.db` і `data/` у теку встановлення. Деінсталятор і оновлення Inno Setup не чіпають файли, яких не ставив інсталятор, тож дані переживають апгрейди.

### 3. Tesseract за замовчуванням

`configure_tesseract(settings.tesseract_path)`: якщо шлях у налаштуваннях порожній — спробувати `<тека exe>/tesseract/tesseract.exe`; якщо є — використати. Налаштування користувача має пріоритет.

### 4. Збірка та версія

- Оновити `main.spec`: режим onedir (`EXE` без бінарників + `COLLECT`), `datas=[("resources", "resources")]`, ім'я `ALU Gauntlet Helper`, іконка як зараз.
- Єдине джерело версії: `alu_gauntlet_helper/version.py` з `APP_VERSION = "x.y.z"`; Inno-скрипт отримує її через `/D` параметр або препроцесор.
- Скрипт збірки `scripts/build_installer.ps1`: `pyinstaller main.spec` → `ISCC installer/setup.iss`.

### 5. Inno Setup скрипт (`installer/setup.iss`)

- `AppId` — фіксований GUID (щоб апгрейди ставали поверх).
- `PrivilegesRequired=lowest`, `DefaultDirName={localappdata}\Programs\ALU Gauntlet Helper`.
- Файли: вміст `dist/ALU Gauntlet Helper/*` рекурсивно + `installer/tesseract/*` у `{app}\tesseract`.
- Ярлики: меню «Пуск», опціонально робочий стіл.
- Опціональна задача «Запускати при старті Windows» — ярлик у `{userstartup}` з аргументом `--minimized`.
- `[Run]` — запустити застосунок після встановлення (опціонально, postinstall).
- Перед установкою/деінсталяцією — закрити запущений екземпляр (`CloseApplications=yes`); деінсталятор видаляє лише встановлені файли, тож `app.db` і `data/` у теці застосунку зберігаються.

## Обробка помилок

- Tesseract не знайдено ні в налаштуваннях, ні в бандлі — поточна поведінка (розпізнавання недоступне, повідомлення в налаштуваннях).

## Тестування

- Юніт-тести на chdir-логіку та фолбек Tesseract на бандл (monkeypatch `sys.frozen`/`sys.executable`).
- Ручна перевірка: збірка інсталятора, встановлення, перший запуск (створення `app.db` у теці застосунку), копіювання наявних даних, повторне встановлення поверх, деінсталяція (дані лишаються).

## Поза скоупом
