# Дизайн: підтримка мови застосунку (Eng/Ukr)

Дата: 2026-06-27

## Мета

Додати локалізацію **інтерфейсу застосунку** (UI) з двома мовами — English та
Українська. Це окремий механізм від наявної мови **контенту гри** (`game_lang`,
en/ru — назви карт/треків). Назви авто завжди залишаються англійською.

## Вимоги (з постановки)

1. У налаштуваннях — випадаючий список із двома пунктами **Eng / Ukr**, що
   змінює мову застосунку (UI-рядки).
2. Контент гри не залежить від мови UI: назви карт/треків керуються своїм
   налаштуванням (`game_language`, en/ru), назви авто — завжди англійською.
3. Зміна мови застосунку застосовується після **перезапуску**. Поряд із
   налаштуванням — статичний приглушений підпис, що пояснює це. Без модалки.
4. При першому запуску, якщо мова застосунку ще не задана, визначити мову
   системи й зберегти: `uk`/`ru` → `uk`, інакше → `en`.

## Архітектура

### Модуль `alu_gauntlet_helper/ui_lang.py`

Новий модуль поряд із `game_lang.py`, той самий легкий патерн (процес-глобальна
змінна, без важких імпортів і циклічних залежностей).

```python
EN = "en"
UK = "uk"

_language = EN

TRANSLATIONS = {
    "en": { "settings.save": "Save", ... },
    "uk": { "settings.save": "Зберегти", ... },
}

def set_ui_language(language: str) -> None: ...   # нормалізує до EN/UK
def current_ui_language() -> str: ...
def t(key: str) -> str: ...                        # рядок поточної мови
```

Відмінності від `game_lang`:
- **Без listeners / live-перемикання.** Мова UI застосовується лише на старті;
  зміна потребує перезапуску. Це навмисно — не треба перебудовувати всі віджети.
- `t(key)`: повертає рядок поточної мови; якщо ключа немає в `uk`, фолбек на
  `en`-рядок; якщо й там немає — повертає сам `key` (видно недопереклад, нічого
  не «зникає»).

Ключі — у форматі `"<область>.<назва>"`, напр. `"tab.races"`, `"settings.save"`,
`"tray.show"`. Області групуються за екранами/вкладками.

### Налаштування — `services/settings.py`

Нове поле в моделі `Settings`:

```python
# мова інтерфейсу застосунку; "" = ще не визначено (тригер автодетекту)
app_language: str = ""   # "" | "en" | "uk"
```

Окреме від наявного `game_language` (контент, en/ru). Порожній дефолт — ключовий:
саме він тригерить автодетект системної локалі при першому запуску.

### Автодетект + застосування на старті — `main.py`

До створення `QApplication` (порядок як зараз для `game_lang`):

1. Прочитати `settings`.
2. Якщо `settings.app_language` порожнє:
   - визначити локаль через `QLocale.system().name()` (повертає напр. `"uk_UA"`,
     `"ru_RU"`, `"en_US"`; працює без інстансу QApplication);
   - перші 2 символи: `uk`/`ru` → `uk`, інакше → `en`;
   - зберегти у `settings.app_language` (`APP_CONTEXT.settings.save`).
3. `ui_lang.set_ui_language(settings.app_language)`.

Розташування: поряд із наявним блоком
`game_lang.set_game_language(settings.game_language)` у `main()`.

### UI налаштувань — `views/settings_tab.py`

- Новий `QComboBox` «Application language» з пунктами **Eng / Ukr**
  (`UI_LANGUAGE_OPTIONS = [("Eng", ui_lang.EN), ("Ukr", ui_lang.UK)]`), угорі
  форми (над/поряд із наявним «Game language»).
- Під ним — статичний приглушений `QLabel` із підказкою про перезапуск
  (текст сам локалізований через `t(...)`). Видимий завжди.
- `refresh()`: вибрати поточне значення `settings.app_language` (фолбек на
  перший пункт).
- `on_save()`: записати `settings.app_language = combo.currentData()`. **Не**
  викликати `set_ui_language` наживо (на відміну від `game_lang`, яка
  застосовується одразу) — мова UI підхопиться при наступному запуску.

### Локалізація рядків UI — `views/`

Замінити захардкоджені англомовні літерали на `t("...")` в усьому UI за один
прохід (увесь UI одразу):

- `main_window.py` — назви вкладок, заголовок вікна, tray-меню;
- `settings_tab.py`, `races_tab.py`, `tracks_tab.py`, `cars_tab.py`,
  `capture_tab.py`, `car_selection_tab.py`;
- `views/components/` — діалоги/кнопки (Save/Cancel/Delete тощо).

Поза локалізацією (без змін):
- назви карт/треків — лишаються на `game_lang`;
- назви авто — завжди англійською;
- технічні/діагностичні рядки, що вже українською в коментарях/статусах —
  переносимо у словник лише якщо вони видимі користувачу.

## Потік даних

```
старт → settings.get()
      → app_language порожнє? → QLocale.system() → uk/ru→uk, else en → save
      → ui_lang.set_ui_language(app_language)
      → QApplication → MainWindow: усі віджети читають t(key) при побудові
```

Зміна мови в налаштуваннях: save у БД → підпис «застосується після
перезапуску» → при наступному старті нова мова підхоплюється автоматично.

## Обробка помилок / крайові випадки

- Невідоме/некоректне значення `app_language` у БД → `set_ui_language` нормалізує
  до `EN`.
- Відсутній ключ перекладу → фолбек en → сам ключ (нічого не падає).
- `QLocale.system().name()` без локалі → порожній/`"C"` → перші 2 символи не
  `uk`/`ru` → `en` (безпечний дефолт).

## Тестування

- `ui_lang`: `set_ui_language` нормалізує значення; `t()` повертає правильну мову,
  фолбек en, фолбек на ключ.
- Логіка автодетекту (чиста функція `system_to_ui_language(locale_name) -> str`):
  `"uk_UA"→uk`, `"ru_RU"→uk`, `"en_US"→en`, `""→en`, `"de_DE"→en`. Винести в
  тестовану функцію, щоб не залежати від реальної системної локалі.
- Settings: дефолт `app_language == ""`; збереження/читання значення.

## Поза обсягом (YAGNI)

- Live-перемикання мови UI без перезапуску.
- Автовизначення `game_language` (контенту) із системної локалі — постановка
  стосується лише мови застосунку.
- Третя+ мова UI, плюралізація, форматування дат/чисел під локаль.
